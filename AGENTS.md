# Repository Instructions

## Repository architecture

- This is a Pulumi bridge for the Descope Terraform provider.
- `provider/resources.go` is the main bridge configuration: package metadata, token mapping, aliases, and language-specific SDK settings belong there.
- `provider/shim/shim.go` adapts the upstream Terraform Plugin Framework provider for the Pulumi bridge.
- `provider/go.mod` and `provider/shim/go.mod` pin the upstream provider, Pulumi bridge, and Go dependencies.
- `provider/cmd/pulumi-resource-descope/` contains the provider entry point and generated schema artifacts.
- `provider/cmd/pulumi-tfgen-descope/` contains the schema and SDK generator entry point.
- `sdk/` contains generated Node.js, Python, Go, .NET, and Java SDKs.
- `examples/` contains the cross-language integration examples exercised by `make test` and CI.

## Tooling and configuration

- Run `mise install` before building. Tool versions are managed by `.config/mise.toml` plus repository overrides in the root `mise.toml`.
- Never edit `.config/mise.toml` directly; it is generated by ci-mgmt. Put repository-specific tool overrides in the root `mise.toml`.
- `.ci-mgmt.yaml` is the source of truth for generated workflows and repository-management files.
- Provider credentials are supplied through `DESCOPE_PROJECT_ID` and `DESCOPE_MANAGEMENT_KEY`. Never hardcode, print, or commit credentials.

## Generated files

- Never edit generated outputs manually. Do not use `apply_patch`, formatters, scripts, or direct writes against generated files.
- This includes everything under `sdk/`, generated documentation under `docs/`, the provider schema, generated GitHub workflows, and any file whose header says it is autogenerated or will be overwritten.
- Change the source configuration or provider source, then regenerate outputs only through the repository Make targets:
  - `make generate` for all generated artifacts.
  - `make generate_sdks` or `make generate_<language>` for SDKs.
  - `make schema` or `make tfgen` for the provider schema.
  - `make build_registry_docs` for registry documentation.
  - `make ci-mgmt` for generated CI, tooling, and repository-management files.
- After changing generator inputs, run the relevant Make target twice and confirm the second run produces no additional diff.

## Common workflows

- For bridge behavior or metadata changes, edit `provider/resources.go` or the provider shim, then run `make schema`, `make generate_sdks`, and `make build_registry_docs` as applicable.
- For an upstream Terraform provider upgrade, use the generated `upgrade-provider` workflow or the standard update-provider tooling. Review Go dependency changes, regenerate the schema and every SDK, and inspect the schema-check result for breaking changes.
- For CI changes, edit `.ci-mgmt.yaml` or root tooling inputs, run `make ci-mgmt`, and commit the generated results together with their source change.
- Use language-specific targets for focused work: `make generate_nodejs`, `make build_python`, `make install_go_sdk`, and the equivalent targets for `nodejs`, `python`, `dotnet`, `go`, and `java`.
- `make build` builds the provider and all SDKs. Run it before `make test`.

## Validation

- Go provider changes: run `make test_provider`, `make lint_provider`, and `(cd provider && go mod verify)`.
- Schema or bridge changes: run `make schema` and inspect the schema diff, especially required-property and token changes.
- SDK changes: run the matching `make build_<language>` target; use `make build_sdks` when the change affects every language.
- Full integration validation: run `make build` followed by `make test`. Integration tests require valid Descope credentials.
- Always run `git diff --check` and confirm the worktree contains no accidental build artifacts.

## Change hygiene

- Keep source changes and their generated outputs in the same PR.
- Do not commit local artifacts such as `.make/`, `.pulumi/`, `bin/`, `node_modules/`, `provider/coverage.txt`, or language build directories.
- Do not discard generated diffs merely because they are large. Verify they were produced by the intended Make target and review them for unexpected API or dependency changes.
- Preserve the existing automated schema-check comment on PRs and call out any breaking changes explicitly in the PR description.
